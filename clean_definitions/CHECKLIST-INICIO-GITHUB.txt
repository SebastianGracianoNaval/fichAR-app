2. ESTRATEGIA "DE A POCO"
================================================================================

--- 2.1 Principios ---
- Un feature funcional por PR (o conjunto pequeño)
- Cada PR: tests pasan, lint OK
- Commits atómicos con mensajes claros
- No hacer "big bang"; integración continua

--- 2.2 Orden sugerido ---
1. Fase 0: Setup, estructura, auth (Semana 1)
2. Primer push a GitHub: repo creado, .gitignore, README, estructura vacía
3. Fase 1: Fichaje core (Semanas 2-4) — feature por feature
4. Tags: v0.1.0 al terminar auth, v0.2.0 al terminar fichaje, etc.

--- 2.3 Tamaño de PR ---
- Ideal: 200-400 líneas cambiadas
- Máx razonable: 800-1000 (dividir si es mayor)
- Descripción: qué, por qué, cómo probar

================================================================================
3. GITHUB: ESTRUCTURA RECOMENDADA
================================================================================

--- 3.1 Repo inicial ---
- Nombre: fichar-app o similar
- Descripción: Sistema de fichaje laboral para Argentina
- .gitignore: node_modules, .env, *.lock (o sí lock, según política)
- README.md: qué es, cómo correr, links a docs

--- 3.2 Ramas ---
- main: estable, deployable
- develop: integración (opcional)
- feature/xxx, fix/xxx: ramas de trabajo

--- 3.3 Protección main ---
- Require PR antes de merge
- Require status checks (CI)
- No force push a main

--- 3.4 Qué NO subir ---
- .env, .env.local, secrets
- build/, dist/, .dart_tool/
- Supabase: service_role key nunca en repo
- Credenciales, API keys

================================================================================
5. CI BÁSICO (GitHub Actions)
================================================================================

Desde Fase 0, configurar:

- Lint: bun run lint, flutter analyze
- Tests: bun test, flutter test (cuando existan)
- Build: flutter build apk --debug (smoke test)
- Trigger: push a main, PR a main

Archivo: .github/workflows/ci.yml (crear en Fase 0)
