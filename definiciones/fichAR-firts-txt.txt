Esta es la hoja de ruta definitiva para fichAR. Este documento está diseñado bajo una mentalidad de arquitectura de software senior, priorizando la seguridad, la optimización extrema y el cumplimiento legal de la reforma laboral argentina.1. Perfil del Proyecto y Personalidad de DesarrolloNombre del Proyecto: fichAR (Gestión y Administración de Cumplimiento Laboral).Filosofía: "Código de Grado Bancario, Interfaz de Grado Humano".Entorno de Desarrollo: Debian 12 (Linux).Gestor de Paquetes: Bun (por su velocidad superior y compatibilidad con scripts de shell).Estándar de Calidad: Clean Architecture + DDD (Domain Driven Design).2. Stack Tecnológico (Optimización y Multiplataforma)CapaTecnologíaJustificaciónFrontend (UI)Flutter 3.xÚnico código base para iOS, Android (viejos y nuevos), Web, Windows, macOS y Linux. Renderizado nativo.BackendGo (Golang)Rendimiento de bajo nivel, manejo de hilos (Goroutines) para procesos pesados y binarios pequeños.Base de Datos / AuthSupabase (PostgreSQL)No queda chico. Soporta RLS (Row Level Security) a nivel de base de datos, vital para la seguridad.Gestor de PaquetesBunEjecución de tests y scripts de despliegue hasta 10 veces más rápido que NPM.Integracionesn8n / WebhooksFacilidad para conectar con cualquier software externo (Jira, SAP, etc.) con desarrollo mínimo.3. Estructura de Roles y PermisosEmpleado (User): Fichaje, visualización de su propio banco de horas, solicitud de licencias, visualización de tareas y noticias legales ("Sabías que").Gerente (Supervisor): Visualización de equipos a cargo, aprobación de licencias, alertas de incumplimiento de descanso (Art. 190) de su equipo.Empleador / Admin: Configuración total del sistema, gestión de lugares de trabajo, reportes masivos, creación de cuentas batch.Auditor (Opcional): Acceso de solo lectura a logs inmutables para peritajes judiciales.4. Arquitectura y Seguridad (Prioridad #1)Para que fichAR sea válida en un juicio laboral, la seguridad debe ser técnica y procedimental:Inalterabilidad (Hashing): Cada registro de entrada/salida genera un Hash SHA-256 único vinculado al registro anterior (Cadena de bloques simplificada). Si un administrador edita una hora, el hash se rompe.Firma Digital: Uso de biometría (FaceID/Huella) para firmar el "Cierre de Jornada" semanal.Audit Logs: Registro de IP, dispositivo, versión de la app y timestamp del servidor (no del celular) para cada acción.2FA / MFA: Integración obligatoria con Google Authenticator para roles de Admin.5. Definición de Pantallas y FuncionalidadesA. App Mobile (iOS/Android - Optimización "Low-End")Dashboard Duolingo Style: Pantalla limpia con un botón central de "Fichar". Barra de progreso del "Banco de Horas".Background Service: Uso de Background Fetch en iOS y WorkManager en Android. Consumo mínimo de batería (solo despierta por geofencing o notificación push).Geofencing: El botón de fichar solo se habilita si el GPS está dentro del radio definido por el empleador (ej. 100m de la oficina).Modo Offline: Permite fichar sin internet; el registro se guarda cifrado localmente y se sincroniza al recuperar conexión con un timestamp firmado.B. App Desktop & WebPanel de Reportes: Generación de XLSX con filtros avanzados:Índice de Presentismo.Ranking de Horas en Banco (Riesgo de pago de extras).Alertas de fatiga (Art. 190).Importador Masivo: Drag & drop de archivos XLSX/CSV para crear 50+ cuentas de una vez o definir perímetros de trabajo hibrido.6. Flujo de Casos Edge (Casos Críticos)Fichaje Remoto no autorizado: El sistema bloquea el fichaje y envía notificación al gerente: "Empleado intentó fichar fuera de zona".Solicitud de Licencia: El empleado sube PDF/Foto de certificado médico. El sistema aplica OCR para extraer fechas y pre-aprobar el período.Cambio de Dispositivo: Si un empleado intenta loguearse en un celular nuevo, requiere validación por correo y código del dispositivo anterior o del Admin.7.
