3. FLUJO DE DATOS PRINCIPAL
================================================================================

--- 3.1 Fichaje ---
Cliente → POST /api/v1/fichajes (body: tipo, lat?, long?, lugar_id?, idempotency_key)
  → Middleware: Auth, validar rol, validar zona (si CFG-005)
  → Service: calcular hash, insertar, disparar webhook si CFG-034
  → Response: { id, timestamp_servidor, ... }
  → Realtime: broadcast a canal org_{org_id} para actualizar dashboard

--- 3.2 Login ---
Cliente → POST /api/v1/auth/login (email, password)
  → Supabase Auth o custom
  → Validar empleado no despedido
  → Validar límite dispositivos (CFG-004)
  → Generar JWT con org_id, role, branch_id?
  → Log: login

--- 3.3 Export reporte ---
Cliente → POST /api/v1/reportes/export (filtros)
  → Validar permisos, filtros
  → Si filas <= umbral: generar sync, responder con URL
  → Si filas > umbral: encolar job, responder { job_id, status: "processing" }
  → Job: generar XLSX, subir a Storage, enviar email con link
  → Log: reporte_exportado

4. EVENTOS Y REALTIME
================================================================================

Canal Supabase Realtime: org_{org_id}

| Evento        | Payload                    | Suscriptores              |
|---------------|----------------------------|---------------------------|
| fichaje       | { user_id, tipo, ts }      | Dashboard empleado        |
| alerta        | { empleado_id, tipo }      | Supervisor, Admin         |
| licencia      | { id, estado }             | Empleado, Supervisor      |

Para añadir evento: 1) Definir payload. 2) Emitir desde service. 3) Suscribirse en cliente.

================================================================================
5. CAPAS Y DEPENDENCIAS
================================================================================

API (packages/api):
  - NO importa Flutter
  - SÍ importa shared (tipos, constantes)
  - SÍ usa Supabase client (server-side)
  - Orden: routes → services → lib (DB, auth)

Mobile/Web (apps/*):
  - SÍ importa shared
  - NO importa api directamente (solo HTTP)
  - Provider/Riverpod para estado global
  - Feature-first o layer-first: elegir uno y mantener

7. ESCALABILIDAD HORIZONTAL
================================================================================

- API: stateless. Múltiples instancias detrás de load balancer.
- Supabase: gestiona escalado. Connection pooling por defecto.
- Jobs: un worker o cola (Bull, Inngest) para reportes async.
- Para añadir worker: definir job type, handler, encolar desde API.

================================================================================
8. DESKTOP: TAURI vs ELECTRON
================================================================================

Recomendación: Tauri (menor uso de recursos).
- Instalador: 2-10 MB vs 80-150 MB (Electron)
- RAM idle: 30-50 MB vs 150-300 MB
- Startup: 0.3-1 s vs 1-3 s

Ver OPTIMIZACION-RECURSOS-RED.txt para detalles.
